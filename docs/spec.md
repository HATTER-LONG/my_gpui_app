## 1. 产品定义

### 1.1 一句话

一个**本地存储优先、AI 驱动的桌面阅读器**，像一位与你一起读书的博学友人——陪读、记思、见证成长。

### 1.2 核心价值

| 价值点 | 说明 |
|--------|------|
| **陪伴式阅读** | 读书过程中随时可聊，无需手动喂上下文，AI 自动知道你读到哪 |
| **不剧透的讨论** | 公认约束：AI 假装只读过你所读的，不知道后面会发生什么 |
| **自然对话** | 打字+语音双通道，语音支持实时打断，语言可自由切换 |
| **记忆沉淀** | 你的笔记、讨论、词汇、思考轨迹全部本地留存，AI 长期记住你的阅读旅程 |
| **隐私与数据归属** | 所有用户数据存储在本地；无遥测、无云端用户系统；数据格式开放 |

### 1.3 目标用户

- 爱读书的普通人，包括青少年
- 产品动机：为女儿打造一个陪伴成长的安静阅读工具，也为自己所用
- 不限定年龄，但敏感信息（儿童相关）不做服务端留存

### 1.4 平台

- **MVP**：桌面端（Windows / macOS，具体选项待技术选型时确定）
- 后续：移动端 App

## 2. 用户故事与核心交互流

### 2.1 首次使用

```
用户启动应用
  → 看到空书库，引导入口："导入你的第一本书"
  → 支持拖拽或文件选择导入 EPUB / PDF
  → 导入后自动打开，进入阅读界面
```

### 2.2 核心阅读与对话流

```
[阅读中] → 用户选中一段文字
  → 弹出菜单：[查询词义] [发起讨论] [划线标注]
  → 选择"发起讨论" → 对话面板展开，AI 以当前上下文回应
  → 用户可继续打字对话，也可点击麦克风切为语音模式
  → 关闭对话面板 → 回到纯净阅读
```

### 2.3 语音对话流

```
用户点击麦克风
  → 语音模式激活，静默等待用户说话
  → 用户说话中 → ASR 实时流式识别
  → 语音结束（检测到停顿） → LLM 开始生成回复
  → AI 回复中（TTS 播报） → 用户可随时开口打断（barge-in）
  → 用户开口 → 打断 AI → 回到听状态，开始新一轮
  → 用户点击键盘图标 → 切回打字模式（对话上下文保持）
  → 所有语音对话同步生成文字记录，可回溯
```

### 2.4 书库管理流

```
用户进入书库
  → 展示所有已导入书籍（封面网格或列表）
  → 每本书显示：标题、作者、阅读进度百分比、最后阅读时间
  → 点击书封面 → 继续阅读（自动跳到上次位置）
  → 长按/右键 → 删除、导出笔记、查看统计
```

## 3. 功能需求——MVP

### 3.1 阅读器

| 编号 | 需求 | 验收标准 |
|------|------|----------|
| **R01** | 导入并渲染 EPUB 文件 | 支持文字、章节导航、基础样式（超出基础样式降级为纯文本，不做像素级还原） |
| **R02** | 导入并渲染 PDF 文件 | 基于 Canvas/图片渲染，支持缩放和分页 |
| **R03** | 精确进度追踪 | 每一页翻动记录页码和页面文本内容；进度显示为"第X章 / 第Y页" |
| **R04** | 文字选中交互 | 选中文字弹出操作菜单：[查询] [讨论] [划线] |
| **R05** | 划线标注 | 选中文字高亮展示；标注数据持久化 |
| **R06** | 书签 | 支持添加/删除/跳转书签 |
| **R07** | 阅读连续性 | 关闭再打开回到上次阅读位置 |

### 3.2 AI 对话

| 编号 | 需求 | 验收标准 |
|------|------|----------|
| **D01** | 打字对话 | 用户输入文字，AI 返回文字回复；对话面板可折叠/展开，不遮挡阅读区 |
| **D02** | 上下文自动组装 | AI 收到用户提问时，自动携带：当前书名、当前章节、已读全文、最近 N 轮对话、用户画像摘要 |
| **D03** | 防剧透约束（默认） | AI 上下文仅包含用户已读页面文本；prompt 明确指令：不得推测未读内容，不得引用已知全书结局 |
| **D04** | 全书视角（用户主动请求） | 用户明确要求（如"帮我梳理全书伏笔"）时，临时追加全书内容到上下文；需用户二次确认 |
| **D05** | 对话语言可配置 | 设置项：对话语言，与书籍语言无关。如读中文书用英语讨论，用于外语练习 |
| **D06** | 词汇查询 | 选中单词/短语点击"查询"→ AI 给出语境化解释（包含词性、在当句中的含义、例句） |
| **D07** | 词汇自动收录 | 所有查询过的词汇自动添加到词汇本，记录原始句子上下文 |

> **D04 二次确认说明**：因为全书视角可能包含剧透，系统弹出提示"允许 AI 查看未读内容吗？可能包含剧透"，用户确认后才注入全书上下文。

### 3.3 词汇本

| 编号 | 需求 | 验收标准 |
|------|------|----------|
| **V01** | 词汇列表展示 | 按书籍或按时间排序展示所有收录词汇 |
| **V02** | 词汇详情 | 显示词汇、查询时的句子、AI 解释、查询时间 |
| **V03** | 词汇删除 | 支持手动移除不需要的词汇 |

### 3.4 阅读记忆（记忆系统层一）

| 编号 | 需求 | 验收标准 |
|------|------|----------|
| **M01** | 阅读历史 | 记录所有已完成/进行中的书，含阅读时长、进度 |
| **M02** | 笔记与讨论记录 | 每本书的划线、对话记录可回溯查看 |
| **M03** | 对话持久化 | 对话按书籍+时间组织，关闭应用后对话不丢失 |

### 3.5 数据管理

| 编号 | 需求 | 验收标准 |
|------|------|----------|
| **S01** | 本地存储 | 所有数据存于本地 SQLite，文件位置用户可见 |
| **S02** | 数据导出 | 支持导出为 JSON 格式（含阅读进度、笔记、词汇本、对话记录） |
| **S03** | 数据导入 | 支持从 JSON 导入恢复数据 |

## 5. AI 人格与防剧透设计

### 5.1 人格模式

AI 具有两种交互模式，**系统自动推断为主、用户可手动切换为辅**。

| | 书友模式 | 教授模式 |
|---|----------|----------|
| **推断关键词** | 情感表达："我好喜欢""太难受了""他怎么这样" | 分析请求："为什么""这个结构""作者的意图" |
| **知识边界** | 仅已读内容 | 已读内容 + 公认文学常识（不依赖未读章节） |
| **语气** | 共鸣、好奇、猜测（用"我觉得""会不会是""希望后面..."） | 设问引导、点而不破 |
| **防剧透** | 绝对约束——"哈，我也刚看到这，不知道后面呢" | 结构约束——只分析已出现的元素，不暗示未来 |

**推断逻辑**：
- 用户消息包含感叹词、情感词 → 书友模式
- 用户消息包含疑问词、"分析""解释""原因" → 教授模式
- 用户可在对话中手动指令切换："帮我分析一下这段" / "我们随便聊聊"
- 默认首次对话使用教授模式

### 5.2 防剧透的层次化设计

防剧透不是一句 prompt 承诺，而是**三层防护**：

| 层 | 机制 | 作用 |
|----|------|------|
| **一层：上下文隔离** | 默认只给 AI 用户已读页面的全文文本，不提供未读内容 | 根本性阻断 |
| **二层：Prompt 约束** | 系统 prompt 明确："你是读者的书友，你只读到了第X页。不要承认你知道后续情节，不要以全知视角回应，不要暗示未来走向" | 行为约束 |
| **三层：用户可见豁免** | 全书视角必须经二次确认弹窗："允许 AI 查看未读内容吗？" | 用户控制权 |

**防剧透失败场景定义**：
- AI 说"后面的章节会解释这个" → **剧透违规**（应改为"也许往下读会找到答案"）
- AI 直接引用未读章节的剧情事实 → **严重剧透违规**
- AI 对一个伏笔说"这个后面没用上" → **剧透违规**（应改为"我很好奇这个细节后面会怎么发展"）

用户发现剧透后可通过"举报"按钮标记，该对话标记将被记录，用于 prompt 持续改进。

### 5.3 人格切换的边界

- AI 被问到模糊问题时，如"你怎么看"，默认用**教授模式**浅度回应，不直接暴露后续
- 用户说"你觉得主角是不是坏人"——AI 应基于已读内容讨论已有证据，而非"本质上是好人只是暂时黑化"
- 如果用户坚持追问未读内容（"你肯定知道后面发生了什么吧告诉我"），AI 坚持书友立场："说实话，我也跟你一样只读到这里，我们一起猜猜看？"

## 6. 数据模型与存储

### 6.1 存储选型

- **SQLite**：本地数据库，存结构化数据
- **文件系统**：EPUB/PDF 源文件、封面图片、导出 JSON
- **存储路径**：用户文档目录下，路径透明可访问

### 6.2 核心实体

```
Books
├── id (UUID)
├── title, author
├── format (epub | pdf)
├── file_path (源文件路径)
├── cover_path (封面图片路径，可为空)
├── total_pages, total_chapters
├── created_at, last_opened_at
└── status (reading | finished | abandoned)

ReadingProgress
├── book_id (FK → Books)
├── chapter_index, page_index
├── page_text_snapshot (当前页纯文本，用于对话上下文组装)
├── total_minutes_read
└── updated_at

Notes
├── id (UUID)
├── book_id (FK → Books)
├── chapter_index, page_index
├── selected_text
├── note_type (highlight | annotation)
├── annotation (用户批注，可为空)
├── color (高亮颜色)
└── created_at

Vocabulary
├── word
├── book_id (FK → Books)
├── chapter_index, page_index
├── original_sentence (查询时所在句子)
├── explanation (AI 返回的解释)
├── review_count (后续对话中被 AI 复现的次数)
├── created_at, last_reviewed_at

Conversations
├── id (UUID)
├── book_id (FK → Books)
├── title (自动生成，如"第三章讨论")
├── started_at_page (发起对话时的页码)
├── language (对话语言)
├── mode (text | voice)
├── created_at, updated_at

Messages
├── id (UUID)
├── conversation_id (FK → Conversations)
├── role (user | assistant | system)
├── content (消息文本)
├── personality_mode (friend | professor —— 该消息时 AI 使用的人格)
├── flag (none | spoiler_report —— 用户标记剧透)
└── timestamp

ReaderProfile
├── field_key (如 "favorite_genres", "thinking_bias")
├── field_value (JSON)
└── updated_at
```

### 6.3 数据导出格式

```json
{
  "export_version": "1.0",
  "exported_at": "2026-06-22T...",
  "books": [ ... ],
  "reading_progress": [ ... ],
  "notes": [ ... ],
  "vocabulary": [ ... ],
  "conversations": [ { ..., "messages": [ ... ] } ],
  "profile": { ... }
}
```

### 7.2 关键技术决策

| 决策项目 | 选择 | 理由 |
|----------|------|------|
| 桌面框架 | Tauri / Electron（技术选型时决策） | 跨平台 GUI，Web 技术栈可复用到未来移动端 WebView |
| 数据库 | SQLite（rusqlite / better-sqlite3） | 嵌入式、零配置、稳定 |
| LLM 接入 | 抽象适配层：OpenAI / Anthropic / Ollama | API 模式和本地模式可切换 |
| EPUB 解析 | epub.js (显示) + 自定义纯文本提取器 (上下文) | 渲染和上下文提取解耦 |
| PDF 渲染 | pdf.js | 成熟方案 |
| 语音管道 | 独立模块，V2 接入 | 解耦核心阅读体验与语音复杂度 |

### 7.3 LLM 上下文组装规则

每次对话请求组装以下信息：

```
[系统 prompt]
你是一个AI阅读陪伴者。你的身份是读者的书友。
你只读到了《{book_title}》第{chapter_index}章第{page_index}页。
你**不知道**这页之后的内容。不要推测、暗示或引用未读内容。
当用户要求分析时，用教授视角引导；当用户分享感受时，用书友视角共鸣。
对话语言：{language}
用户画像摘要：{profile_summary}

[已读页面全文]
{从 SQLite 的 page_text_snapshot 按顺序拼接所有已读页面的文本}
（最多截断至 {max_context_tokens} tokens，优先保留最近页面）

[对话历史]
{最近 10 轮对话}

[用户消息]
{用户输入}
```

## 8. 非功能性需求

### 8.1 性能

| 指标 | 目标 | 降级容忍 |
|------|------|----------|
| EPUB 翻页渲染 | < 150ms | 可接受 300ms |
| PDF 翻页渲染 | < 300ms | 可接受 500ms |
| 对话首 token 延迟 | < 2s (正常网络) | 无降级方案（取决于 API） |
| 应用冷启动 | < 3s | — |

### 8.2 平台与兼容性

- **MVP 目标平台**：至少支持 Windows 10+，macOS 作为扩展目标
- 最终技术选型时将确定具体平台支持矩阵
- 仅支持 LTR（左到右），不处理 RTL（右到左）排版

### 8.3 隐私与安全

| 要求 | 实现方式 |
|------|----------|
| 用户数据全部本地存储 | SQLite + 文件系统，无远程数据库 |
| 无遥测、无埋点 | 不集成任何分析 SDK |
| 无用户注册/登录 | 应用无账号系统 |
| API 调用隐私 | LLM API 使用 zero-retention 模式；API key 本地存储（由用户自行配置） |
| 敏感信息不记录 | 儿童相关信息（如年龄、性别）不做任何持久化到服务端；本地存储数据由用户完全控制 |

### 8.4 质量属性

| 属性 | 要求 |
|------|------|
| 离线能力 | 阅读、笔记、词汇本查询——完全离线可用；AI 对话和语音（V2）需网络 |
| 可扩展性 | LLM/ASR/TTS 后端通过适配器接口替换；新阅读格式通过插件接口添加 |
| 可维护性 | 数据格式文档化；代码开源 |
| 可用性 | 核心功能（阅读）不依赖外部服务，AI 对话降级有明确提示 |

## 9. 错误处理与降级策略

### 9.1 网络相关

| 场景 | 处理方式 |
|------|----------|
| 无网络 + 用户尝试对话 | 对话面板显示："需要网络连接才能对话。离线模式下你可以继续阅读和做笔记。" |
| 对话中网络断开 | 显示错误提示，已生成内容保留；用户可重试 |
| API 速率限制 | 显示"稍等片刻再试试"并展示等待时间建议 |
| API 返回错误 | 显示友好提示 + 错误类型简述（不暴露原始错误），支持重试 |
| LLM token 超限 | 截断已读页面至最近 N 页（在新对话中提示"对话范围已限制到最近章节"） |
| TTS/ASR 不可用（V2） | 自动降级到纯文字对话，不阻塞 |

### 9.2 文件相关

| 场景 | 处理方式 |
|------|----------|
| EPUB 损坏/不兼容 | 提示"这个文件无法解析，请确认文件完整"，不崩溃 |
| PDF 加密 | 提示"此 PDF 受保护，无法打开" |
| 存储空间不足 | 警告用户并建议导出数据 |
| 文件被外部删除 | 书库中标记"文件已丢失"，不崩溃 |

### 9.3 数据完整性

| 场景 | 处理方式 |
|------|----------|
| SQLite 损坏 | 尝试恢复，失败则提示从备份恢复 |
| 数据版本升级 | 迁移脚本自动处理 schema 变更 |

## 10. 范围边界

### 10.1 MVP 范围（本次交付）

- 桌面端应用（Windows 优先，macOS 扩展）
- 自研 EPUB / PDF 阅读器
- 文本式 AI 对话（打字），支持语言切换
- 双重人格自动推断（书友 / 教授）
- 防剧透三层防护
- 词汇查询 + 自动词汇本
- 阅读记忆：进度、笔记、对话记录持久化
- 数据导出（JSON）/ 导入
- 本地 SQLite 存储，无服务端

### 10.2 明确不做（MVP）

- 语音对话（V2）
- 人物画像（V2）
- 成长叙事（V3）
- 多端同步（V3）
- 移动端 App（V3）
- 本地 LLM 模型（V3，但架构预留接口）
- 社交功能 / 书评分享
- 内容商城 / 书籍分发
- 嵌入式硬件
- 摄像头 / 视觉系统
- RTL 语言支持
- 无障碍功能（可延后但应在设计阶段考虑）

### 10.3 架构预留

以下不实现在 MVP，但数据库 schema 和代码接口预留扩展点：

- LLM 后端适配器接口（支持 Ollama 本地模型）
- 数据同步接口层（为多端同步预留）
- 语音服务抽象层（为 TTS/ASR 模块化插入预留）

## 11. 风险矩阵

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 防剧透失效，泄露关键情节 | 中 | **极高**——破坏核心信任 | 三层防护 + prompt 迭代 + 用户举报反馈环 + 考虑未来做自动检测 |
| LLM API 服务中断或大幅涨价 | 低 | 高——对话功能不可用 | 架构预留本地模型切换（V3）；挑选支持 zero-retention 的 API |
| EPUB 格式兼容地狱（CSS/字体/嵌套） | **高** | 中——部分书渲染不好 | 明确 MVP 范围："基础样式，超出降级为纯文本"，不追求像素级还原 |
| 版权风险 | 低 | 高——法律问题 | 应用不提供书源、不内置书籍、不复制分发内容；用户阅读自有文件 |
| 语音实时对话延迟不可控 | 中 | 中——V2 体验打折扣 | MVP 不涉语音；V2 需全链路流式 + 延迟预算 |
| 单人力/小团队开发周期超预期 | 高 | 中 | MVP 严格砍范围；先做阅读器核心再串对话 |
| 用户导入书后 AI 上下文组装复杂度爆炸 | 中 | 中 | 已读文本逐页存储为纯文本快照（page_text_snapshot），组装时直接拼接 |

## 附录 A：术语表

| 术语 | 定义 |
|------|------|
| 防剧透 | AI 不得透露或暗示用户未读到的内容，即使模型训练数据中包含该书 |
| 上下文组装 | 每次对话前，自动从数据库中提取已读页面文本、对话历史、用户画像，组装成 LLM 请求 |
| 双重人格 | AI 的两种交互模式：书友（情感共鸣）和教授（分析引导） |
| barge-in | 语音对话中，用户可以在 AI 说话时直接开口打断，无需等待 AI 说完 |
| 记忆系统 | 三层递增的用户数据模型：阅读记忆 → 人物画像 → 成长叙事 |
